# 기여 가이드 (CONTRIBUTING)

> BOS 4.0 디자인 시스템에 기여하는 방법.

이 문서는 **디자이너, 개발자, PM 모두**를 위한 가이드입니다.
변경 제안이나 새로운 컴포넌트가 필요하다고 느끼셨다면 이 가이드를 따라주세요.

---

## 🙋 누가 어떻게 기여하나요

### 디자이너

**작은 수정** (오타, 라벨 변경, 색상 미세 조정):
1. Figma에서 직접 수정
2. 해당하는 문서 파일(`docs/*.md`)도 같이 업데이트
3. GitHub Desktop에서 커밋 메시지 작성 후 push

**큰 변경** (새 컴포넌트 추가, 토큰 시스템 수정):
1. GitHub Issue 생성 → 팀과 논의
2. Figma에서 디자인 + 문서 업데이트
3. PR(Pull Request) 생성

### 개발자

**버그 리포트**:
- GitHub Issue에 다음 정보 포함:
  - 재현 단계
  - 예상 동작 vs 실제 동작
  - 브라우저/OS 정보
  - 스크린샷

**제안/개선**:
- GitHub Issue에 `enhancement` 라벨로 생성
- 간단한 것이면 바로 PR도 환영

### PM / 기획자

**새 화면 패턴 요청** (예: "협업 중 실시간 커서 표시 패턴 필요"):
1. GitHub Issue 생성, `pattern-request` 라벨
2. 유사 사례 (다른 서비스의 예시) 링크 첨부
3. 디자이너와 함께 논의 후 `docs/04-layout-patterns.md`에 추가

## ✅ 커밋 메시지 컨벤션

간단한 규칙을 따르면 나중에 변경 이력 찾기 쉬워요.

```
[카테고리]: [간단한 설명]

[자세한 내용 — 선택사항]
```

**카테고리 예시**:
- `토큰` — 토큰 추가/변경
- `컴포넌트` — 컴포넌트 추가/수정
- `패턴` — 레이아웃 패턴 추가/수정
- `문서` — 가이드라인 문서 변경
- `사이트` — site/ HTML 파일 변경
- `수정` — 버그, 오타, 일관성 문제 수정

**좋은 예**:
```
토큰: text-fg-brand-strong Dark mode brand-100 → brand-300 변경

다크모드에서 active 메뉴 가독성 향상을 위한 조정.
WCAG 대비비 3.2 → 4.8 개선.
```

**나쁜 예**:
```
update
fix
수정
```

## 🚫 하지 말 것

- ❌ **Figma에서 새 색상 만들기** → 기존 토큰 조합 고려, 꼭 필요하면 논의 후 추가
- ❌ **시멘틱 토큰 이름 함부로 바꾸기** → 사용하는 모든 곳에 영향. 이름 변경은 PR 필수
- ❌ **`main` 브랜치에 직접 push** (팀원 2명 이상일 때) → PR을 통해 리뷰
- ❌ **시각 카탈로그만 수정하고 토큰 CSS 안 바꾸기** → 불일치 발생, 둘 다 수정
- ❌ **변경 이력 업데이트 없이 끝내기** → 각 문서 하단의 "📝 변경 이력"에 한 줄 추가

## 📢 변경 공지

큰 변경사항(브레이킹 체인지, 새 컴포넌트 출시 등)은:

1. GitHub에서 Release 생성 (태그 + 릴리스 노트)
2. 팀 Slack/이메일로 공유
3. 연관된 프로젝트 저장소에 마이그레이션 가이드 링크
