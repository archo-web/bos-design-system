# ux-writing.md — BOS 4.0 UX 라이팅 가이드

> 이 문서는 BOS 4.0 시스템의 **한국어 UX 라이팅 표준**을 정의합니다.
> 같은 기능이라도 문구 하나로 사용자 경험이 크게 달라집니다.

---

## 🎯 BOS 4.0 라이팅 톤앤매너

**3단어로 정의하면**: **명확하게 / 짧게 / 친절하게**

- **명확하게**: 모호한 단어 금지, 구체적인 동사 사용
- **짧게**: 같은 의미면 더 짧은 쪽 선택, 군더더기 제거
- **친절하게**: 존댓말 통일, 사용자 책망 금지, 다음 행동 안내

**피해야 할 톤**
- ❌ 너무 캐주얼 ("앗! 뭔가 잘못됐어요 😅")
- ❌ 너무 격식 ("당해 작업이 정상적으로 처리되었음을 알려드립니다")
- ❌ 책망조 ("입력값이 잘못되었습니다. 다시 입력하세요")
- ❌ 책임 회피 ("시스템 오류로 인해...")

## 1. 어조 통일

### 1.1 존댓말 — "~하세요" 톤

| 상황 | ✅ 사용 | ❌ 피하기 |
|---|---|---|
| 안내 | "프로젝트 이름을 입력하세요" | "프로젝트 이름을 입력해 주십시오" |
| 안내 (긴 표현) | "이메일을 확인해 주세요" | "이메일 확인이 필요합니다" |
| 결과 알림 | "저장되었습니다" | "저장이 완료되었습니다" |
| 질문 | "정말 삭제하시겠습니까?" | "삭제하실래요?" |

**원칙**: 가능하면 "~하세요" 형태, 부탁하는 뉘앙스가 필요하면 "~해 주세요".

### 1.2 시제

- **결과 알림**: 과거형 "저장되었습니다" / "삭제되었습니다"
- **진행 상태**: 진행형 "저장 중입니다" / "업로드하고 있습니다"
- **안내**: 명령형 "입력하세요" / "선택하세요"

### 1.3 띄어쓰기 & 맞춤법

자주 틀리는 것들:

| ✅ 올바름 | ❌ 잘못됨 |
|---|---|
| 로그인 | 로그-인 |
| 로그아웃 | 로그-아웃 |
| 회원가입 | 회원 가입 (둘 다 허용이지만 통일) |
| 비밀번호 | 패스워드 / 비번 |
| 다음 단계 | 다음단계 |
| 더 보기 | 더보기 |
| 새로 만들기 | 새로만들기 |
| 일치하지 않습니다 | 일치하지않습니다 |

**팀 컨벤션** (정한 것은 끝까지 통일)
- "비밀번호" 사용 (패스워드 ✗)
- "이메일" 사용 (메일 ✗)
- "삭제" 사용 (제거 ✗)

## 2. 버튼 라벨

### 2.1 기본 원칙

- **동사로 시작**
- **2~4자가 이상적**, 5자 넘으면 재검토
- **무엇을 할 것인지 명확히**

### 2.2 표준 라벨 사전

| 의미 | ✅ 사용 | ❌ 피하기 |
|---|---|---|
| 신규 생성 | 추가 / 만들기 / 등록 | 신규 / 새로 / 추가하기 |
| 저장 | 저장 | 저장하기 / 등록 |
| 수정 진입 | 수정 | 편집하기 / 변경 |
| 삭제 | 삭제 | 제거 / 지우기 |
| 확인 (일반) | 확인 | OK / 알겠습니다 |
| 취소 | 취소 | 닫기 / 그만 |
| 다음 단계 | 다음 | 다음으로 |
| 이전 단계 | 이전 | 뒤로 / 이전으로 |
| 제출 | 제출 | 보내기 / 전송 |
| 검색 실행 | 검색 | 찾기 / 조회 |
| 초기화 | 초기화 | 리셋 / 비우기 |
| 적용 | 적용 | 반영 |
| 닫기 | 닫기 | X (✗) |

### 2.3 컨텍스트별 추천

**파괴적 액션** — 일반적인 "확인" 대신 **실제 동작 명시**

| ❌ Bad | ✅ Good |
|---|---|
| 정말 삭제하시겠습니까? [확인] [취소] | 정말 삭제하시겠습니까? [삭제] [취소] |
| 결제하시겠습니까? [확인] [취소] | 결제하시겠습니까? [결제하기] [취소] |

**구체성** — 무엇을 다음에 할지 명시

| ❌ Bad | ✅ Good |
|---|---|
| [확인] | [프로젝트 만들기] |
| [제출] | [신청서 제출] |
| [완료] | [등록 완료] |

## 3. 폼 라벨 & 헬퍼 텍스트

### 3.1 라벨 (BOS 4.0 `Label` 컴포넌트)

- **명사형**: "이메일" / "프로젝트 이름" / "전화번호"
- **물음표 ✗**: "이메일을 입력하세요" 말고 "이메일"
- **콜론 ✗**: "이메일:" 말고 "이메일"

### 3.2 Placeholder

- **예시 형식 안내**: 라벨을 반복하지 않음

| ❌ Bad | ✅ Good |
|---|---|
| 라벨: 이메일 / placeholder: 이메일을 입력하세요 | 라벨: 이메일 / placeholder: example@company.com |
| 라벨: 전화번호 / placeholder: 전화번호 | 라벨: 전화번호 / placeholder: 010-1234-5678 |
| 라벨: 날짜 / placeholder: 날짜 선택 | 라벨: 날짜 / placeholder: YYYY-MM-DD |

### 3.3 헬퍼 텍스트 (BOS 4.0 `Helper` 컴포넌트)

- 라벨로 다 설명할 수 없는 제약/안내
- 입력 전 보여주는 정보

```
비밀번호
[                              ]
8자 이상, 영문/숫자/특수문자 포함
```

### 3.4 필수/선택 표시

- **필수 필드**: 라벨 옆에 빨간 별표 `*` (`text-fg-danger`)
- **선택 필드 표시는 선택적**: 모든 필드가 필수에 가까우면 별표 생략 가능
- 둘 다 섞이면 **소수 쪽**을 표시

## 4. 에러 메시지

### 4.1 황금 공식

```
무엇이 + 왜 + 어떻게
```

### 4.2 좋은 예시

| ❌ Bad | ✅ Good |
|---|---|
| 입력 오류 | 이메일 형식이 올바르지 않습니다. @ 기호를 포함해 주세요. |
| 비밀번호가 틀렸습니다 | 비밀번호가 일치하지 않습니다. 다시 확인해 주세요. |
| 네트워크 오류 | 네트워크 연결이 불안정합니다. 잠시 후 다시 시도해 주세요. |
| 권한 없음 | 이 작업을 수행할 권한이 없습니다. 관리자에게 문의하세요. |
| 파일 업로드 실패 | 파일 크기가 10MB를 초과합니다. 더 작은 파일을 선택해 주세요. |

### 4.3 사용자 책망 금지

| ❌ 책망 | ✅ 중립/도움 |
|---|---|
| 잘못된 이메일을 입력했습니다 | 이메일 형식이 올바르지 않습니다 |
| 비밀번호를 잘못 입력했습니다 | 비밀번호가 일치하지 않습니다 |
| 필수 항목을 입력하지 않았습니다 | 이메일을 입력해 주세요 |

### 4.4 시스템 책임 회피 금지

| ❌ 회피 | ✅ 책임 + 해결 안내 |
|---|---|
| 시스템 오류가 발생했습니다 | 데이터를 불러올 수 없습니다. 잠시 후 다시 시도해 주세요. |
| 서버에 문제가 있습니다 | 일시적인 오류가 발생했습니다. [다시 시도] |
| 알 수 없는 오류 | 오류가 발생했습니다 (코드: ERR_500). 문제가 계속되면 문의해 주세요. |

## 5. 빈 상태 메시지

### 5.1 3가지 구성 요소

```
1. 상태 (왜 비어있는지)
2. 가치 (무엇이 가능한지)
3. 행동 (어떻게 시작하는지)
```

### 5.2 상황별 예시

**처음 사용 (Onboarding)**
```
✅ 등록된 프로젝트가 없습니다
   첫 프로젝트를 만들어 팀과 함께 협업해 보세요
   [+ 프로젝트 만들기]
```

**검색 결과 없음**
```
✅ '디자인 시스템'에 대한 검색 결과가 없습니다
   다른 키워드로 검색하거나 필터를 조정해 보세요
   [검색 초기화]
```

**필터 결과 없음**
```
✅ 조건에 맞는 항목이 없습니다
   필터를 변경해서 다시 시도해 보세요
   [필터 초기화]
```

**모두 처리됨 (긍정적)**
```
✅ 확인할 알림이 없습니다
   모두 잘 처리되었어요
```

### 5.3 피해야 할 빈 상태

| ❌ Bad |
|---|
| "데이터가 없습니다" (단독) |
| "비어있음" |
| "결과 없음" |
| 빈 화면 그대로 두기 |

## 6. 성공 메시지 (Toast)

### 6.1 짧고 명확하게

| ❌ 길다 | ✅ 짧다 |
|---|---|
| 프로젝트가 성공적으로 저장되었습니다 | 프로젝트가 저장되었습니다 |
| 변경사항이 정상적으로 반영되었습니다 | 변경사항이 저장되었습니다 |
| 파일 업로드가 완료되었습니다 | 파일이 업로드되었습니다 |

**원칙**: "성공적으로", "정상적으로", "완료"는 거의 항상 군더더기.

### 6.2 행동 가능한 알림 (Toast Variant: Avatar & button)

```
✅ 프로젝트 '알파'가 삭제되었습니다  [실행 취소]
✅ 5개 항목이 보관함으로 이동되었습니다  [되돌리기]
✅ 초대장을 보냈습니다  [초대 현황 보기]
```

가능하면 다음 행동을 제공해서 사용자 흐름 유지.

## 7. 확인 다이얼로그 문구 (Modal Type: Popup)

### 7.1 표준 형식

```
[제목 — 질문 형식, 명확하게]
[본문 — 무엇이 일어나는지, 영향 범위]

[취소]  [실제 동작 명시 버튼]
```

### 7.2 좋은 예시

```
✅ 프로젝트를 삭제하시겠습니까?
   '프로젝트 알파'를 삭제하면 연결된 12개 작업도
   함께 삭제됩니다. 이 작업은 되돌릴 수 없습니다.
   
   [취소]  [삭제]
```

```
✅ 변경사항을 저장하지 않고 나가시겠습니까?
   저장하지 않은 변경사항이 사라집니다.
   
   [계속 편집]  [저장 안 함]  [저장 후 나가기]
```

### 7.3 나쁜 예시

```
❌ 정말로요?
   삭제할 수 없습니다.
   [확인]  [취소]
```
- 모호함, 영향 미설명, "확인"이 무엇을 확인하는지 불분명

## 8. 마이크로카피 모음

### 8.1 자주 쓰는 문구

| 상황 | 표준 문구 |
|---|---|
| 로딩 (일반) | 불러오는 중입니다 |
| 로딩 (저장) | 저장 중입니다 |
| 로딩 (긴 작업) | 처리 중입니다... 잠시만 기다려 주세요 |
| 자동 저장됨 | 자동으로 저장되었습니다 |
| 미저장 변경 알림 | 저장되지 않은 변경사항이 있습니다 |
| 새로고침 권유 | 새로운 업데이트가 있습니다. [새로고침] |
| 더 많은 결과 | 더 보기 (12개 더) |
| 비활성 사유 | 권한이 없어 사용할 수 없습니다 |
| 다크 모드 전환 | 라이트 모드 / 다크 모드 |

### 8.2 숫자 표기

| 상황 | 표기 |
|---|---|
| 천 단위 구분 | 1,234 (콤마) |
| 큰 숫자 | 1.2만 / 12.3만 (한국식) |
| 백분율 | 75% (붙여 쓰기) |
| 통화 | ₩12,000 또는 12,000원 (둘 중 하나로 통일) |
| 날짜 | 2026.04.20 또는 2026년 4월 20일 |
| 시간 | 오후 3:45 또는 15:45 (둘 중 하나로 통일) |
| 상대 시간 | 방금 / 5분 전 / 어제 / 3일 전 / 2026.04.20 |

### 8.3 0과 1의 처리

- "0개" 대신 "없음" 또는 빈 상태로 표시
- "1개"는 그냥 "1개" (영어처럼 단/복수 구분 없음)

## 9. 한영 혼용

### 9.1 영문 그대로 쓰는 경우

- 고유명사: Slack, GitHub, Figma
- 약어: API, URL, ID, SSO, CSV, OTP
- 일반화된 IT 용어: 로그인, 로그아웃, 클릭, 다운로드

### 9.2 번역하는 경우

| 영문 | ✅ 한국어 | ❌ 한영 혼용 |
|---|---|---|
| Save | 저장 | Save |
| Delete | 삭제 | Delete |
| Edit | 수정 | Edit |
| Settings | 설정 | Settings |
| Profile | 프로필 | Profile |
| Dashboard | 대시보드 | Dashboard |
| Notification | 알림 | Notification |

**판단 기준**: 사용자가 일상적으로 한국어로 말하는 단어는 한국어로.

## 10. 빠르게 쓰는 체크리스트

문구를 쓸 때 자문할 것:

- [ ] 더 짧게 쓸 수 있는가?
- [ ] 사용자가 무엇을 해야 하는지 명확한가?
- [ ] 사용자를 책망하고 있지 않은가?
- [ ] 시스템 책임을 회피하고 있지 않은가?
- [ ] 다른 화면의 같은 액션과 라벨이 일치하는가?
- [ ] "성공적으로", "정상적으로" 같은 군더더기가 있는가?
- [ ] 영어 직역처럼 어색하지 않은가?
- [ ] 띄어쓰기가 팀 컨벤션과 일치하는가?
- [ ] 다크 모드에서 읽기 어려운 표현은 없는가? (대부분 색상 문제, 라이팅 문제는 드물지만 점검)
